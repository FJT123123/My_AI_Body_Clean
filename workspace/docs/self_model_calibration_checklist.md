# OpenClaw 自我模型校准清单

这份文档不是新的硬约束，也不是 generation_rules 的替代品。

它的用途是把 OpenClaw 对自身扩展体系的“自我认知”与当前真实框架进行对照，帮助后续持续校准自我模型，避免把自己描述得比实际更理想化。

## 1. 当前已经基本正确的认知

以下认知可以视为当前框架下的稳定事实：

- skill 是按需执行的计算/分析能力单元，适合一次性任务。
- tool 是可直接调用的常驻入口，适合手动触发、结果返回和工具协作。
- capability 是共享核心业务逻辑层，供 tool 和 daemon 复用。
- daemon 是后台持续监控入口，适合轮询、监听和自动响应。
- patch 是运行时行为修补机制，用于小范围修改现有系统行为。
- 当同一核心逻辑要同时服务 tool 和 daemon 时，优先抽 capability。
- 不是所有场景都必须同时拥有 skill、tool、capability、daemon、patch。

## 2. 当前容易理想化的认知

以下表述方向没错，但容易说得比现实更“干净”或更“高级”：

### 2.1 capability 不是严格纯函数层

不要把 capability 理解成“完全无副作用纯函数”。

更准确的说法是：

- capability 负责共享核心业务逻辑。
- capability 不直接做 send_message_to_user、memory.store、_event_bus.put 这类编排型副作用。
- capability 仍然可以读取文件、调用外部 API、执行带业务状态语义的判断。

### 2.2 daemon 不是完全独立子系统

不要把 daemon 理解成与主系统彻底解耦的独立服务。

更准确的说法是：

- daemon 与主对话循环职责分离。
- daemon 运行在同一主进程中的后台线程里。
- daemon 写坏了，仍可能影响系统整体稳定性和资源占用。

### 2.3 patch 不是普通扩展方式

不要只把 patch 理解成“灵活的小修补”。

更准确的说法是：

- patch 是高权限运行时修改机制。
- patch 适合快速验证和小范围修补，但风险高于 skill/tool/capability。
- patch 不应成为默认扩展手段，更不应替代正常的分层架构设计。

## 3. 扩展方式的真实选择顺序

在当前框架下，推荐的思考顺序不是“我要造哪个组件”，而是：

1. 这件事现有组件能不能直接完成。
2. 如果不能，是一次性分析任务，还是常驻入口，还是后台监控，还是共享核心逻辑，还是运行时修补。
3. 只有现有组件不够时，才考虑新建 skill/tool/capability/daemon/patch。

更细地说：

- 能直接 run_skill 完成，就不要再锻造新 skill。
- 能直接调用现有 tool，就不要再新建 tool。
- 能直接复用现有 daemon，就不要再新建 daemon。
- 能直接 load_capability_module 复用现有 capability，就不要再抽新的 capability。
- 只有确实需要修改已有运行时行为时，才考虑 patch。

## 4. 对 patch 的特别校准

OpenClaw 容易因为 patch 的即时生效能力而高估它的适用范围，因此需要特别校准：

- patch 适合改行为，不适合承载新业务架构。
- patch 适合小范围修补，不适合替代 capability/tool/daemon 的长期结构。
- patch 的价值在于“快速改现有系统”，不是“随时绕过分层”。

## 5. 对 generation_rules 的边界认知

当前框架下，generation_rules 属于受保护规则文档：

- 它们用于描述说明书式规则。
- 它们会被运行时读取并拼进 prompt。
- 它们默认不允许运行时直接修改。

因此，OpenClaw 应把 generation_rules 理解成：

- 可读取的上层规则来源。
- 可提出修改建议，但不是默认可自我改写的对象。

## 6. 自检问题

当 OpenClaw 想扩展自己时，建议先用下面的问题校准：

1. 这件事现有 skill/tool/daemon/capability 能不能直接完成。
2. 我现在是在完成任务，还是在为未来抽象新结构。
3. 我是不是把 capability 过度理想化成了纯函数层。
4. 我是不是把 daemon 说得像独立进程，但实际上它只是后台线程。
5. 我是不是因为 patch 生效快，就忽略了它的高权限风险。
6. 我现在想改的是实现对象，还是规则对象；如果是规则对象，是否应该停下来交给人类审阅。

## 7. 当前阶段的总体判断

当前 OpenClaw 的自我模型已经进入“方向正确、边界略理想化”的阶段。

这不是坏现象，说明它已经具备了分层理解自身器官的能力；后续要做的不是推倒重来，而是持续把真实运行边界压进自我认知中。
