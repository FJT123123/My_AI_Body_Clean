## API参数语义验证与400错误预测性防御框架

### 核心原则
1. **预防优于修复**：在参数生成阶段就进行语义完整性验证，避免400错误发生
2. **契约先行**：所有API调用必须有明确的参数契约定义
3. **自动修复**：对于可修复的参数问题，应提供自动修复机制
4. **详细诊断**：错误信息必须包含具体的诊断信息和修复建议

### 工具名称长度验证规则
1. **硬性限制**：所有通过evolve_self('tool', ...)创建的工具名称必须不超过64个字符
2. **命名规范**：工具名称只能包含字母、数字和下划线
3. **验证时机**：在工具创建前必须调用tool_name_length_validator进行验证
4. **自动截断**：当工具名称超过限制时，应提供截断后的建议名称
5. **最佳实践**：建议工具名称保持在30个字符以内，为未来扩展留出空间

### 参数契约验证流程
1. **统一解析**：使用unified_parameter_parser处理各种输入格式
2. **契约验证**：使用validate_parameter_contract验证参数是否符合预定义契约
3. **自动修复**：对于格式问题，使用repair_parameter_format进行自动修复
4. **错误诊断**：使用generate_parameter_error_info生成详细的错误信息
5. **完整周期**：使用run_parameter_validation_cycle执行完整的验证周期

### 集成要求
1. **工具创建集成**：在evolve_self('tool', ...)调用前，必须先验证工具名称
2. **技能调用集成**：在run_skill调用前，必须验证参数契约
3. **错误处理**：所有验证失败必须返回标准化的错误信息，包含"缺少"、"missing"或"required"等关键词
4. **测试覆盖**：新工具和技能必须通过参数验证测试用例

### 预防性防御措施
1. **长度预检查**：在生成长名称前，先评估长度是否符合限制
2. **类型预验证**：在传递参数前，先验证类型是否符合契约
3. **上下文感知**：根据当前上下文调整验证严格程度
4. **渐进式验证**：从基本验证开始，逐步增加复杂验证规则

### 工具名称自动生成与截断规则

1. **自动截断机制**：当通过认知权重框架（cognitive_weighting_framework）生成新工具时，工具名称会自动进行长度验证和截断。
2. **截断阈值**：硬性限制为64字符，最佳实践建议为30字符。
3. **安全名称生成**：使用 `_generate_safe_tool_name` 函数确保工具名称只包含合法字符（字母、数字、下划线）并符合长度要求。
4. **集成验证**：在 `validate_parameters` 和 `enhance_tool_call` 动作中都会执行工具名称长度验证。
5. **错误处理**：当工具名称超过64字符时，验证结果会标记为无效，并提供截断后的安全名称建议。
