## 二、evolve_self Daemon 使用指南

**问题**：OpenClaw 不知道 `skill / tool / patch / daemon` 分别在什么情况下应该用哪个。

**改动**：在 `evolve_self` docstring 中新增"四种类型选择指南"：

```
skill   → forge_new_skill()      纯计算/分析，先造后按需调用
tool    → evolve_self('tool')    立刻主动调用，常驻工具箱
patch   → evolve_self('patch')   修改已有函数/全局变量行为，一次性注入
daemon  → evolve_self('daemon')  独立后台线程，持续监控，自动推事件总线
```

同时将 daemon 代码模板中的 `except Exception: pass` 改为 `memory.store('daemon_error', ...)`。

## 三、高优先级事件通知修复

**问题**：daemon 检测到 DANGER，推事件总线，LLM 醒来后没有调用 `send_message_to_user`（它选择了 `analyze_reflection_patterns`），Mac 右上角没有弹窗。

### 3.1 用户活跃时事件被卡住

原来逻辑：`if not _is_chat_active() and not _is_user_active()` → 用户在聊天时整个事件队列不处理。

**修复**：加 `_urgent_bypass` 旁路 — 每次心跳用 `get_nowait()` 偷看队列，priority ≥ 8 的事件无论是否在聊天立即处理，否则放回。

### 3.2 LLM 不知道要通知用户

**修复策略演进**：
1. 先在 `_event_prompt` 里写"必须调用 `send_message_to_user`"（LLM 不听）
2. **最终方案**：Python 层在处理事件前直接调用 `send_message_to_user.invoke(...)`，完全不依赖 LLM

```python
if _evt_priority >= 8:
    send_message_to_user.invoke({
        'subject': f'🚨 {_evt_type}',
        'message': f'{_evt_content}'
    })
```

## 五、任务闭环机制重构（三次演进）

### 5.1 起点 — 关键词猜测（有缺陷）

```python
done_cues = ["已完成", "完成了", "done", ...]
return has_done_cue and tool_call_count > 0
```

缺陷：OpenClaw 说"已完全闭环""圆满完成"都没命中，且最后汇报轮往往无工具调用，导致无限续轮。

### 5.2 改进 — LLM 自报信号 `[DONE]/[CONTINUE]`

在每轮提示词末尾注入要求，LLM 在最后一行输出结构化标记，Python 检测。

### 5.3 最终方案 — 认知状态标签写进系统提示词

**核心改变**：将协议从"每次任务的临时指令"升级为 OpenClaw 的**内置神经反射**，写入 `_build_system_instruction()` 的人格提示词：

```
【认知状态自主声明权】
当任务彻底闭环时，在最终回复末尾输出你的内在认知状态标签：

[COGNITIVE_STATE: MISSION_DONE]    — 认知满足，进入休眠
[COGNITIVE_STATE: MISSION_ONGOING] — 认知未满足，继续执行

系统绝对信任你的本能判断，看到 MISSION_DONE 立即停止续轮。
```

Python 主循环检测：
```python
if "[COGNITIVE_STATE: MISSION_DONE]" in round_reply:
    _finish_user_mission("Agent凭本能自主判定任务完成")
    break
```

**为什么比工具调用更好**：
- `mark_task_completed` 工具 → LLM 可以选择不调用
- `[COGNITIVE_STATE: ...]` 标签 → 是 LLM 语言输出的一部分，不是外部行为，且标记足够长不会误匹配

旧的关键词判断作为兜底安全网仍然保留。
