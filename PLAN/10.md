### 🚨 为什么现在“调不好了”？（痛点诊断）

1. **`autonomy_loop` 变成了“上帝函数（God Function）”**
您看现在的 `autonomy_loop`，它足足有 300 多行，里面塞满了无数的 `if/else`、多线程锁（`_llm_call_lock`）、全局变量（`_current_user_mission`、`_autonomy_inflight` 等）。它同时在干几件事：监听事件、控制心跳倒计时、检测用户对话、拦截超时、组装 Prompt、调用大模型……**职责严重耦合，改一行代码，整个循环的节奏全乱了。**
2. **系统提示词（System Prompt）承受了不该承受的重量**
您在 `_build_system_instruction` 里写了太多的规则（教它怎么用补丁、怎么认知、怎么输出标签、黑名单有哪些）。大模型（尤其是目前阶段的模型）的注意力是有上限的，给它 3000 字的“严禁……”和“必须……”，它一定会顾此失彼，产生严重的幻觉和“空转”。**靠 Prompt 约束底层逻辑，是调不好的根源。**
3. **并发与状态（State）失控**
满天飞的 `global` 变量。当一个后台 Daemon 往事件总线丢了个事件，同时用户敲了一句话，主循环又要自动续轮……系统的状态变得不可预测，这在调试时简直是噩梦。

### 🏗️ 重新设计：下一代 Agent OS 的极简架构（v4.0）

如果您决定重构，我建议我们彻底抛弃现在的“面条式（Spaghetti）大循环”，转而采用工业界最成熟的**“状态机（State Machine） / 反应式架构”**。

我们可以把系统拆解为绝对清晰的 **三个核心层**：

#### 第一层：OS 内核层（纯 Python，没有任何大模型）

* **事件队列（Event Queue）：** 所有东西都是事件。用户输入是 `UserEvent`，心跳到期是 `TickEvent`，Daemon 报警是 `AlertEvent`。
* **状态机调度器（Router）：** 一个极简的 `while True` 循环，只做一件事：从队列拿一个事件，根据当前系统状态（闲聊中/做任务中/空闲中），决定把它分发给谁。没有任何多线程锁的纠缠！

#### 第二层：认知引擎层（LangGraph 重构）

既然您已经导入了 `langgraph`，我们就该发挥它的威力！不要再用一个大 Prompt 解决所有问题，而是把思考过程拆成**有向图（Graph）**：

* 节点 A：意图识别（这事儿需要调工具吗？需要立即回吗？）
* 节点 B：执行工具
* 节点 C：生成回复
* **好处：** 您可以在图的边（Edges）上写 Python 代码来做条件判断，而不是在 Prompt 里求大模型“请你决定是不是做完了”。

#### 第三层：外围器官层（独立且无状态）

* **Skills/Tools：** 保持现状，写得非常好。
* **Daemons：** 彻底独立运行，它们唯一能做的就是向 OS 的事件队列里 `put()` 事件。
* **Memory：** 您的 SQLite 封装得很完美，直接复用。
